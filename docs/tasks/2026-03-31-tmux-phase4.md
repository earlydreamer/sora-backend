# Phase 4: tmux 오케스트레이션 선택적 고도화

**상태**: codex-ready
**연결 이슈**: #9

## 목표

tmux 오케스트레이션 3단계 이후 운영 편의성과 안정성을 높이는 선택적 고도화 항목 4개를 구현한다.

## 범위

### 포함 항목

1. **`dashboard` 스크립트** (신규): `watch` + ANSI color 기반 TUI 상태판
   - 모든 워커의 상태를 한 화면에 표시
   - `dashboard` 단독 실행 → 실시간 갱신 (`watch -n 5`)
   - `--once` 옵션 → 단일 스냅샷 출력 후 종료
   - `--json` 옵션 → JSON 출력 (프로그래매틱 소비용)

2. **`enqueue-worker` 스크립트** (신규): worker 우선순위 큐
   - `.orchestrator/queue.json`에 task를 우선순위별로 쌓기
   - `enqueue-worker codex routes task-1 --priority 1 --spec docs/tasks/...`
   - `spawn-worker --from-queue` 옵션: 큐에서 다음 항목을 꺼내 워커 생성
   - 우선순위: 낮을수록 먼저 실행 (1 = 최고 우선순위)

3. **`spawn-worker` 수정**: pane 기반 로그 뷰 + 재시도 정책
   - `--split-log` 옵션: window를 main pane(70%) + log pane(30%)으로 분할
     - log pane은 `.orchestrator/logs/<worker-id>.log`를 `tail -f`로 표시
   - `--retry N` 옵션: 워커 실패 시 최대 N회 자동 재시작
     - `worker.json`에 `retry_count`, `max_retries` 필드 추가

4. **`recover-session` 수정**: `--auto-fix`에서 failed 워커 자동 재시도
   - 기존: failed 워커를 stale로 표시만 함
   - 변경: `max_retries` > `retry_count`인 failed 워커를 자동으로 재시작
   - 재시작 시 `spawn-worker` 동일 파라미터로 재호출

5. **`lib.sh` 수정**: queue 관련 공통 함수 추가
   - `queue_push()`: 큐에 항목 추가 (jq 기반, 우선순위 정렬)
   - `queue_pop()`: 우선순위 가장 높은 항목 꺼내기
   - `queue_list()`: 전체 큐 JSON 출력

### 미포함 항목

- 중앙 daemon / 메시지 브로커
- 웹 대시보드
- worktree 기반 격리 (별도 단계)

## 구현 세부 사항

### dashboard 스크립트

```bash
#!/bin/bash
# dashboard: TUI status board for tmux orchestration workers

# 기본: watch -n 5로 실행 (실시간 갱신)
# --once: 단일 출력 후 종료
# --json: JSON 출력

# 출력 형식:
# ╔══════════════════════════════════════════╗
# ║  sora-backend 오케스트레이션 상태판       ║
# ║  2026-03-31 15:30:00                     ║
# ╠══════════════════════════════════════════╣
# ║ worker-001  codex  routes  running  ✓   ║
# ║ worker-002  codex  auth    done     ✓   ║
# ║ worker-003  gemini review  failed   ✗   ║
# ╚══════════════════════════════════════════╝
```

### enqueue-worker 스크립트

```bash
# queue.json 구조:
{
  "queue": [
    {
      "id": "q-001",
      "agent": "codex",
      "slug": "routes",
      "task_ref": "task-1",
      "spec_path": "docs/tasks/...",
      "priority": 1,
      "enqueued_at": "2026-03-31T..."
    }
  ]
}
```

### spawn-worker 추가 옵션

```bash
# --split-log: 2 pane 분할
tmux split-window -t "sora-backend:${window_name}" -v -p 30
tmux send-keys -t "sora-backend:${window_name}.bottom" "tail -f '${log_file}'" Enter

# --retry N: worker.json에 필드 추가
"retry_count": 0,
"max_retries": N
```

### recover-session 재시도 로직

```bash
# failed 워커 중 retry 가능한 것 처리
if jq -e '.retry_count < .max_retries' "${worker_json}" > /dev/null 2>&1; then
  # retry_count 증가 후 spawn-worker 재호출
  spawn-worker "${agent}" "${slug}" "${task_ref}" \
    --spec "${spec_path}" \
    --retry "${max_retries}"
fi
```

## Claude 재호출 조건

- queue.json과 worker.json 간 상태 정합성 문제가 발생할 경우
- `--split-log` pane 분할이 특정 tmux 버전에서 동작하지 않을 경우
- 재시도 정책의 race condition이 의심될 경우
